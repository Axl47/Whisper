# WHISPER AGENTS DOCUMENT

## ExecPlans

When writing complex features or significant refactors, use an ExecPlan (as described in `.docs/PLANS.md`) from design to implementation. Write new plans to `.docs/exec/`. If inside Plan Mode, create the plan in a multiline markdown block, and write it after initiating implementation, so you can use the plan to guide your implementation and refer back to it as needed. If outside Plan Mode, you can write the plan directly and refer to it as needed.

## Development Details

Whenever new updates are made, this file (`AGENTS.md`) should be updated with any surprising files not apparent from the codebase that could benefit other developers. Focus on the why and when it could be useful.

- `SourcePackages/checkouts/FluidAudio/Package.swift`: Local SwiftPM checkout settings can be the practical place to patch toolchain compatibility for dependency sources during local builds. This is useful when Xcode starts surfacing Swift concurrency diagnostics inside `SourcePackages/checkouts/FluidAudio/...`, but remember the checkout can be regenerated by package resolution, so upstreaming or pinning the dependency is safer long-term.
- `SourcePackages/checkouts/FluidAudio/Sources/FluidAudio/ASR/Streaming/StreamingAsrManager.swift`: Swift 6.2 can reject actor-held `AsrManager` references with `Sending 'asrManager' risks causing data races` during local builds. This checkout is the practical patch point when the streaming actor is intentionally the synchronization boundary and the package needs a temporary `nonisolated(unsafe)` compatibility fix.
- `Scripts/apply_local_package_patches.sh` and `Scripts/patches/fluidaudio-swift62-streaming-asr.patch`: Use these when a disposable `SourcePackages` checkout still needs a durable local compatibility fix. They let `run.sh` and manual `xcodebuild -clonedSourcePackagesDirPath SourcePackages ...` flows reapply the FluidAudio Swift 6.2 patch after package resolution without vendoring the whole dependency.
- `SourcePackages/checkouts/FluidAudio/Sources/FluidAudio/ASR/AsrManager.swift`: On Xcode beta / Swift 6.2 toolchains, a local sendability compatibility patch here can be enough to unblock this repo's build, but `SourcePackages/` is gitignored. Use `xcodebuild -clonedSourcePackagesDirPath SourcePackages ...` when validating local SwiftPM checkout patches; otherwise Xcode will compile a separate checkout from `DerivedData` and ignore your workspace-local fix.
- `OpenSuperWhisper/Whis/Whis.swift` and `OpenSuperWhisper/Whis/WhisperFullParams.swift`: The local Whisper bridge already exposes token-level timestamps through `fullNTokens`, `fullGetTokenText`, `fullGetTokenData`, and `WhisperFullParams.tokenTimestamps`. This is the first place to look when live streaming starts repeating merged prefixes, because segment timestamps alone are not stable enough for rolling dedup.
- `OpenSuperWhisper/TranscriptionService.swift`: Whisper token timestamps are useful for stability gating, but the tokens themselves are still subword pieces. Live streaming should only commit token-backed text up to a safe word boundary; otherwise partial tokens leak as artifacts like `halluc hallucinations` or `that's's`.
- `OpenSuperWhisper/Utils/FocusUtils.swift`: Live AX insertion cannot trust the element’s current selected range after each write. Keep the insertion range in the session and feed it back into subsequent writes, otherwise some apps will snap the next delta back to the original caret and prepend new chunks at the start.
- `OpenSuperWhisper/Indicator/IndicatorWindow.swift` and `OpenSuperWhisper/TranscriptionService.swift`: The stable version of the Whisper hotkey flow now keeps streaming strictly inside the popup and pastes only once after the post-release batch decode finishes. When adjusting this flow, preserve `transcribeAudio(..., preserveDisplayedText: true)` in the live hotkey branch so the popup keeps the last live preview visible while the final batch pass runs.
- `OpenSuperWhisper/Indicator/IndicatorWindow.swift` and `OpenSuperWhisper/Indicator/IndicatorWindowManager.swift`: The indicator’s transcript area now depends on both SwiftUI layout and AppKit window resizing. If the popup starts clipping or drifting, check the transcript container width in the view and the `resizeWindowToFitContent()` logic together; the panel keeps its top edge anchored and grows downward as transcript text appears.
- `OpenSuperWhisper/Indicator/IndicatorWindow.swift`: The hotkey overlay now has two intentionally matched visual paths: a native Liquid Glass branch gated by `#available(macOS 26.0, *)` using `GlassEffectContainer` / `glassEffect`, and a material fallback for older targets. When tuning the overlay look, keep both branches visually aligned so the popup does not become a different product on older macOS versions.
- `OpenSuperWhisper/Indicator/IndicatorWindow.swift`: The overlay reads best as one primary HUD surface with one calmer transcript panel. Multiple fully outlined nested glass pills made the layout look broken in practice, so if the indicator starts feeling heavy again, reduce the number of bordered sub-surfaces before adding more tint, blur, or chrome.
- `OpenSuperWhisper/Workflows/LiveRecordingCoordinator.swift`: This is now the shared completion path for live microphone recordings from both the main window and the mini recorder. If workflow matching, recording persistence, or paste-vs-command delivery diverges between those entry points, start here before debugging the two callers independently.
- `OpenSuperWhisper/Settings.swift`: The workflows editor intentionally uses draft state inside a sheet instead of binding fields straight into `AppPreferences`. If validation starts feeling inconsistent, check the draft-to-save flow first; invalid intermediate edits are supposed to stay local to the sheet.
- `SourcePackages/checkouts/FluidAudio/Sources/FluidAudio/ASR/Streaming/StreamingAsrManager.swift`: On the current Xcode beta toolchain, local validation may require removing a few `nonisolated(unsafe)` stored-property annotations here before `xcodebuild -clonedSourcePackagesDirPath SourcePackages ...` will compile the app target. Treat it as a local compatibility patch only; `SourcePackages/` is still disposable.

## Sub Agents

Use sub-agents where appropriate to break down complex changes into manageable pieces, and to allow for more focused implementation and testing. For example, if implementing a new feature that requires both backend and frontend changes, you might create separate sub-agents for each layer of the stack, but before then use an exploring agent (or multiple) to get context on the codebase and research the best approaches for the feature, outline the specific steps needed for implementation into a final exec plan, and spin up task subagents that handle the implementation. This allows for more efficient development and testing, as each sub-agent can focus on a specific aspect of the implementation, and can be tested independently before being integrated into the larger codebase.

## Final Output

When asking the user to verify implemented changes, output a checklist they can fill to make sure everything works as intended. Describe what they should see, how it should work, and what they need to manually test. The user will then fill in the checklist and provide feedback on any issues they encounter, which can be used to further refine the implementation.

If the user asked for multiple changes and only some were implemented, make sure to clearly indicate which ones were completed, which ones were not fully realized, and which ones are still pending. For example:

```txt
- [x] Implement app scaffold (completed with basic layout and navigation)
- [~] Implement feature A (stub implementation completed)
- [ ] Implement feature B (pending due to X reason)
```

Include a commit message after each implementation or fix, following the Conventional Commits specifications. If it's a large change, follow this format:

```txt
feat(update): add startup update prompt choices and sectioned changelog pipeline
- feat(update): gate startup updates behind user choice (Yes/No/Remind Later)
- feat(update): persist per-release prompt decisions (ignore until newer, 24h remind-later)
- refactor(update): split updater flow into eligibility check and install phases
- feat(update): parse GitHub release body into sectioned changelog blocks for in-app prompt
- test(update): add updater decision/state-store/changelog parser coverage
- feat(ci): generate release notes sections from commit metadata and publish via body_path
- feat(ci): support multi-section changelog from Conventional Commit lines in commit body
- fix(navigation): clamp bottom navbar sizing to prevent tiny rendering on some phones
- fix(navigation): make top-level tab swipe detection more reliable in Explore
- fix(search): move Explore apply+navigate to app scope to prevent canceled loads on slower devices
- docs(readme): document updater prompt behavior and changelog contract
```
